File Access and Operations

As mentioned previously, files are referred to by their filenames–arbitrary strings–with no concept of multi-component paths or any other file system structure.

Removing and Renaming Files

A file may be removed with the int remove(char const *filename); function, and renamed with the int rename(char const *oldname, char const *newname); function.

Either function returns zero on success, and a non-zero value on failure.

Temporary Files

An unnamed temporary file may be created with the FILE *tmpfile(void); function. This function returns a pointer to the stream (FILE object) that it created. The file created by tmpfile is removed when the program exits.

The char *tmpnam(char *s); function may be used to generate a unique, valid file name that is not the same name as any existing file. If its argument s is a null pointer, tmpnam returns a pointer to the generated filename in an internal static object, which may be modified by a subsequent call to tmpnam. Otherwise, tmpnam stores the result in the object pointed at by s, which must be an array of at least L_tmpnam char, and returns a pointer to that object.

Both functions return a null pointer on failure–typically a result of resource exhaustion.

Note

The tmpnam function is subject to a race condition. Between the time that a filename is generated and the time that the program opens that file, another process could potentially create a file with the same name, leading to conflicts. Safer alternatives are often provided as system-specific functions, such as the POSIX mkstemp function, which atomically creates and opens a temporary file with a unique name.

Opening and Closing Files

Files may be opened using the FILE *fopen(char const *filename, char const *mode) function. The access mode is a string, with the following significance,

Mode string

Explanation

"r"

Open a file for reading

"w"

Truncate (or create) a file for writing

"a"

Open (or create) a file for appending

"r+"

Open a file for read/write

"w+"

Truncate (or create) a file for read/write

"a+"

Open (or create) a file for read/appending

When a file is opened in append mode, the position is always set to the end of the file before a write operation. The meaning of reading on a file opened with "a+" is not well-specified, and varies.

In speech, the modes are typically referred to as “read”, “write”, and “append”, while the modes with a ‘+’ are referred to as “extended read”, “extended write”, and “extended append”.

An additional ‘b’ may be added to any of the above mode strings, such as “rb” or “rb+”. This sets the stream to binary mode; otherwise the stream is in text mode. A binary mode stream treats its contents as a sequence of bytes, while a text mode stream treats its contents as a series of lines, and may discard non-text bytes and whitespace at the end of a line. Binary mode is almost always preferred since it does not discard data.

The returned FILE* points to an opaque data structure, or handle, which is used internally by other stdio functions. On failure, a null pointer is returned.

Standard Streams

The standard streams are already opened and ready to be accessed when a program is invoked, and need not be closed when a program exits. These are accessed via the macros stdin, stdout, and stderr which expand to expressions of type FILE* just as any other stream. Importantly, these are not variables–they are macros–and do not necessarily expand to lvalues that can be directly modified. Attempting to reassign one of the standard streams–e.g. stdin = fopen("input.txt", "r");–will result in undefined behavior.

The FILE *freopen(char const *pathname, char const *mode, FILE *stream) function is provided specifically to handle this situation. Rather than allocate a new FILE object, freopen closes stream and opens the specified file in-place on the given FILE object, as in freopen("input.txt", "r", stdin).

As with fopen, the freopen function returns a pointer to the open stream, or null on failure.

Closing a file

An open file may be closed by passing it as an argument to fclose(FILE *stream). Note that, because streams use buffers, opening a file involves allocating a buffer behind the scenes, and thus can produce memory leaks if the file is not closed before a program terminates!

Stream Status Indicators

Streams have two status indicators–error and end-of-file. Many file i/o operations can fail; for example, several functions return a special (and unfortunately named) EOF macro to indicate failure. Others return null pointers, where a non-null pointer was expected, or return a short item count, when reading or writing a fixed number of items. In any case, none of those functions indicate the cause of the failure, and it must instead be evaluated by querying these status indicators,

  • int feof(FILE *stream); queries the end-of-file indicator, which is set when an i/o function fails due to reaching the end of input.

  • int ferror(FILE *stream); queries the error indicator, which is set when an i/o function fails for another internal reason; for example, exhaustion of resources, parsing error, etc.

  • void clearerr(FILE *stream); unsets both indicators, and is the only mechanism by which these indicators can be reset, once set.

These functions never fail and do not modify errno, so it is safe to check the error indicator on a stream before checking errno for any additional implementation-defined error information.

Buffering

As mentioned previously, standard i/o streams use an i/o buffer to move data between program memory and the external environment in large, efficient chunks.

Buffering Modes

There are three buffering modes that a stream can have–fully buffered, line buffered, and unbuffered. In fully buffered mode, a stream has a fixed-size output buffer which is automatically flushed to external storage whenever it is filled; this is also referred to as block buffering, because data is written in large blocks. Line buffering is the same as full buffering, except that output is also automatically flushed whenever a newline ('n') character is written to the stream. In unbuffered mode, data is written to external storage as soon as it is available.

When a file is opened, as with fopen, it is buffered by default. If it refers to an interactive device, such as a terminal, it is line buffered; otherwise, it is fully buffered. The standard streams stdin and stdout are also line buffered, while stderr is unbuffered, by default.

After a file is opened, but before any other operations have been performed on it, its buffering mode may be modified with the int setvbuf(FILE *stream, char *buf, int mode, size_t size); function.

The mode argument must be either, _IOFBF, _IOLBF, or _IONBF, each of which causes i/o to be fully buffered, line buffered, or unbuffered, respectively. If _IOFBF or _IOLBF is specified, the buf argument should point to a buffer of at least size bytes, which will be used as the buffer. It may also be a null pointer, in which case the library will allocate a suitable buffer on the next i/o operation. The contents of the buffer while it is in use are indeterminate.

Warning

If setvbuf is used to assign a buffer to a stream, that buffer must have a lifetime that extends at least until the stream is closed; otherwise the behavior is undefined.

An additional int setbuf(FILE *stream, char *buf); convenience function is exactly equivalent to,

int setbuf(FILE *stream, char *buf)
{
   setvbuf(stream, buf, buf ? _IOFBF : _IONBF, BUFSIZ);
}

In other words, if buf is null, the stream is set to unbuffered mode. Otherwise, buf must be at least BUFSIZ bytes, and the stream is set to fully buffered mode.

Both functions return 0 on success, non-zero on failure.

Alternating I/O

One important thing to note is that the same buffer is used for both input and output. Stream input methods will cause any buffered output to be flushed, and then the buffer will be filled with input data from the external storage. Similarly, stream output methods will discard any buffered input before placing the output data in the buffer. There is no difference between line and full buffering when the buffer is used for input–it is always completely filled with input data whenever it is depleted.

Because of input prefetching, the real file position can move ahead of the perceived file position. This only becomes a problem if an output method is used after an input method, and data is discarded from the buffer–those discarded bytes represent the difference between the real and perceived file positions.

If the underlying file supports seeking, then the library will update the real file position to the perceived position when switching the buffer from input to output. However, not all file types support seeking, such as a terminal device file, and will result in errors due to the failed seek operations. For this reason, it is generally unsafe to perform an output operation after an input operation on a buffered stream.

Flushing a Buffer

At any time, the contents of a buffer can be explicitly flushed with the int fflush(FILE *stream); function, which returns 0 on success, and non-zero if an error occurs.

Repositioning

Streams support a very basic positioning mechanism, which, as mentioned above, may not be available if the underlying storage does not support seeking.

The long ftell(FILE *stream); function returns the value of the file position indicator, or returns -1 and sets errno to an implementation-defined value on failure.

The int fseek(FILE *stream, long offset, int whence); function sets the file position indicator relative to whence, which may be SEEK_SET–the beginning of the file; SEEK_CUR–the current position; or SEEK_END–the end of the file. It returns non-zero on failure. Text mode streams can only be repositioned with an offset of 0 or with an offset equal to the return value of a previous ftell, with whence set to SEEK_SET.

On POSIX systems, seeking past the end of the file and then performing a write operation will zero-fill the intervening space.

The void rewind(FILE *stream); function seeks to the beginning of a file and clears the error indicator. It is equivalent to (void)fseek(stream, 0L, SEEK_SET); clearerr(stream);.

Note

Some online resources demonstrate querying the size of a file by repositioning to the end of a file, querying the position, and then rewinding the file,

long size = -1;
if (fseek(fp, 0L, SEEK_END) == 0) {
   size = ftell(fp);
   rewind(fp);
}

This is generally a poor programming practice, since the availability of stream positioning is not a valid assumption.

Wide Streams

As an extension to the standard i/o library, streams can support multi-byte, or wide, character encodings such as unicode. The methods for this are exposed through the additional whchar.h header, and most are direct analogues of the byte-oriented methods that will be covered in this module; for example, wchar_t getwchar(void); is the wide-oriented analogue of int getchar(void); byte-oriented method. Wide streams have an associated parse state, so simply using the above repositioning methods could cause the stream to be repositioned to a location where the current parse state is invalid; it is only safe to rewind a wide stream using these methods.

A pair of functions int fgetpos(FILE *stream, fpos_t *pos); and int fsetpos(FILE *stream, fpos_t *pos); are provided as a means to save and then restore a file position and its parse state; both functions return 0 on success, and non-zero on failure. Additionally, errno is set to an implementation-defined positive value on failure. These are the only safe mechanism for seeking to a location that is not the beginning of a wide stream.

Wide streams are an advanced topic that will not be discussed further in this course.
