CS 374 - Operating Systems

Introduction

Recall the original motivations driving the development of operating systems: the desire to execute multiple programs concurrently, and on different types of hardware. Before the advent of the operating system, a human operator (from which the operating system derives its name) was responsible for manually loading and executing individual programs, one at a time. The state of the machine would be reset between each program, and each program had unfettered access to all of the machine’s hardware–meaning each program was also written specifically for a particular hardware configuration and would need to be manually converted, or “ported”, to run on another machine. Computer scientists quickly realised that this model was unsustainable, as entire collections of software would need to be rewritten every time systems were upgraded. Additionally, they recognized the inefficiencies caused by sequential execution of programs; with I/O as the primary execution bottleneck, the CPU sat idle for much of a program’s execution.[1]

These two motivations led to two basic ideas. The first was to develop a collection of primitive functions that could be preloaded into memory before each program was executed–and I mean this in a quite literal sense, as in a stack of punch cards that would be added to the beginning of each program before it was loaded into the punch card reader. These primitive functions would mediate access to hardware through an abstract interface, and programs would call these functions rather than access the hardware directly, themselves. This reduced the size of each program, and also made programs more portable. Migrating to different hardware meant swapping out this “library” of code, without having to modify the actual program itself.

This is what would later develop into the concept of a kernel and its collection of system calls that we have today–all of which are mapped into each process’s address space.[2] Some programs might also depend on other modular collections of code that extend this core set of functionality, and which would be inserted between the core library code and the program’s code. This parallel concept eventually developed into modular, loadable libraries, such as the C standard library, which the operating system also loads into each process’s address space as needed to satisfy that program’s dependencies.

The shift towards programs accessing hardware resources through these abstract interfaces allowed programmers to take the next step towards the genesis of the modern operating system: batch processing. In these systems I/O operations were asynchronous–once the processor initiated an I/O request specialized hardware would handle the rest of the task of moving that data into the desired memory location while the main processor sat idle. The speed of this I/O was orders of magnitude slower than the processor itself, but the program that made the request could not continue until it was completed.

This gave computer scientists an idea–what if instead of sitting idle, we load multiple programs into memory at the same time, and switch between them each time an I/O operation is requested? With enough programs, we could minimize the amount of time that the CPU sits idle. All this required was for programs to support relocation–the ability to execute regardless of which region of memory they were loaded into, enabled through the use of relative, rather than absolute, addressing to access internal functions and data. Some additional data structures were added to the core library code (let’s call it a “kernel” now), which would store the execution state of each program (let’s call this a process). When a process called a kernel I/O function, that function would initiate the request and then, instead of going to sleep, would store the CPU register states (the CPU context) to this internal structure and then load the saved CPU register states of another process and jump to its saved PC (program counter) register location! This “context switching” allowed operators to load multiple programs at once, and would ensure that resources were being used as efficiently as possible.

Later, as computers themselves became more interactive rather than their earlier uses as purely computational machines, the need for responsiveness became apparent–simply switching between processes only when I/O requests occurred could result in an inconsistent user experience. This led to the development of preemptive multitasking–the operating system would set a hardware alarm to trigger an interrupt after a small time slice, just before executing a particular process. When the alarm triggered the interrupt, it would jump into a piece of kernel code, now called the scheduler, which would have the opportunity to perform a context switch. If this could be performed frequently enough, it would give users the illusion of concurrent execution, called “apparent concurrency” (Many years later, multicore CPUs would yield true concurrency!). The scheduler is today a very important component of the modern operating system, and is responsible for carefully balancing responsiveness against the overhead of frequent context switches.

Over time, an additional need for isolation of processes, in order to improve security and system stability, finally led to the matured operating system concepts we work with today–the filesystem with its fine-grained access permissions, virtual memory to isolate processes’ address spaces from one another, and special restricted CPU operating modes, called rings (kernel-mode is ring 0, and user-mode is ring 3). In concert with this, process attributes were added, such as user and group credentials, process hierarchies, and groupings such as process groups and sessions, as well as a rich array of inter-process communication techniques such as signals, pipes, sockets, and so on.

In this module, we will study how the operating system represents and manages processes, including concepts such as process state, execution context, file system resources, credentials, and signals. We will also study how processes are created, made to execute new programs, terminated, and otherwise manipulated and managed from a systems programming perspective.

Each function was assigned a particular number, called its system call number, and this would be passed as an argument to the system call handler. Essentially, this handler function performed a table lookup to jump to the appropriate library function. The address of this system call handler would typically be stored into the processor’s interrupt vector table (IVT) at a particular location (e.g. int 0x80) so that programs could call that function by manually triggering that interrupt.

Conceptually, the entire kernel is mapped, but in modern systems, only a small part of the kernel is mapped into a process’s memory. This portion of code, which contains a system call handler, is used as a “trampoline” that maps the rest of the kernel into memory after entering kernel mode in order to execute a system call. When it is finished, this memory is unmapped again from that process before returning to user mode. This is a feature called “Kernel page-table isolation”, and mitigates the Meltdown <https://nvd.nist.gov/vuln/detail/CVE-2017-5754> security vulnerability.
