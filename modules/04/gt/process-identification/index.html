Process Identification

Processes can be recognized by their Process Identification Number, commonly known as the PID. This is a unique positive integer that is assigned to each process by the kernel. The system call getpid() in C returns the PID associated with the calling process. Every process is also a child of another process. For example, when a user requests to run a program from a shell, the process generated by that program becomes a child of the shell. A process may obtain the PID of its parent process from the system call getppid() (Get Parent Process ID).

Processes have another ID called their group ID. Related processes can be assigned to the same group ID. Usually, a process inherits its group and group id from its parent process. However, the parent process may decide to assign a different group to its children. This is what usually happens with commands in shells. For example, if a shell creates a pipeline of commands, like ls | sort, two processes are created, one for the ls command and another for the sort command, both with different PIDs. Because these processes are related to each other as part of a pipeline, the shell assigns them both to a new process group. This affects the delivery of signals, and other operations, which can be made to apply to entire process groups rather than individual processes. For example, the SIGINT signal that the terminal generates when pressing ctrl-c is sent to the terminal’s foreground process group, so that each process in that group receives the interrupt signal. One job of the shell, when running interactively, is managing which process group is designated as the foreground process group of the terminal, which will receive signals that the terminal generates, and also have permission to read input from the terminal without receiving the SIGTTIN signal. This is called job control, and the shell uses the tcsetpgrp() and tcgetpgrp() system calls for this functionality.

In a process, a system call getpgrp() is used to obtain its own group id. To identify the group id for any other process, one may use the system call getpgid(pid), where the pid parameter contains the PID of the requested process. A process can also change its own group id, or its child’s group id (under certain circumstances), with the setpgid(pid, pgid) system call. The setpgid() system call can be used to join another existing process group, or create a new one, in which case the process ID of the calling process is used as the new group ID that it is now the sole member of.

A session is a set of process groups, all of which are assigned to the same session ID. The session ID is the PID of the process that created the session, known as the session leader. For example, when a shell starts running, it creates a session made out of a single group with a single process, the shell itself, using the setsid() system call. This shell process is the session leader. Every command that runs later on the shell will belong to a group that may be different from the shell group, but they all inherit the same session ID, the session ID of the session leader. The system call getsid(pid) returns the session id of a process with the parameter pid. When this system call is given a pid of zero, it returns the session id of the calling process.

Notice that the same data type pid_t is used for process group IDs and session IDs; this is because both properties are taken from the process ID of the process that creates either a group or session. [3]

[1]

If the PID argument is 0, the caller’s PID is used; if the PGID argument is 0, the PID of the target process is used. If both arguments are 0, a new process group is created with the caller as its only member, and PGID equal to the caller’s PID.

[2]

An additional function, pid_t getpgrp(void) is equivalent to getpgid(0). Extra functions like this are sometimes provided either for legacy reasons, or because the specialized function is often slightly more efficient than the generic version. This is similar to, e.g. printf vs fprintf.

[3]

Be careful not to mix up a process group ID (PGID/PGRP) with a process’s group ID (GID). The former has data type pid_t and refers to the process group which a process belongs to. The latter has type gid_t and refers to the user group ID that are part of the process’s credentials.
